传输层

1. 传输层服务和协议

  • 在不同主机上运行的应用程序进程之间提供逻辑通信
  • 传输层协议运行在端系统上
    • 发送方：将应用信息分解成段，并传递给网络层
    • 接收方：将段重新组装成报文，并传递给应用层
  • 两种传输层协议：TCP和UDP

1.1 传输层和网络层

  • 传输层：进程之间的逻辑通信
    • 网络层服务的依赖和增强
  • 网络层：主机之间的逻辑通信
    • 传输协议提供的服务往往受到底层网络层协议服务模式（延迟或带宽）的限制。但有些服务不包括在内，如可靠的数据传输服务。

1.2 网络层

  • 互联网的网络层协议叫作IP，即网际协议（Internet Protocol）
  • IP 提供主机之间的逻辑通信
  • IP 服务模式是一种尽力而为的传送服务
  • 它不能保证段的有序传送，也不能保证段中数据的完整性
  • IP 被称为不可靠的服务（unreliable service）。

1.3 传输层

  • 可靠、按顺序传输：TCP（Transmission Control Protocol）
    • 数据传输和错误检查
    • 流量控制
    • 拥塞控制
    • 连接设置
  • 不可靠、无序传输：UDP（User Datagram Protocol）
    • 数据传输和错误检查
    • 没有扩展“尽力而为”的IP
  • 两者都无法提供的服务
    • 延迟保证
    • 带宽保证

2. 多路复用和多路分解

  • 在发送端进行多路复用（multiplexing）：处理来自多个套接字的数据，添加传输头。
  • 在接收端进行多路分解（demultiplexing）：使用头信息将收到的段传递给正确的套接字。

2.1 多路分解工作流程

  • 主机接收IP数据报（datagram）
    • 每个数据报有源IP地址、目的IP地址
    • 每个数据报携带一个传输层段（segment）
    • 每个段有源端口号、目的端口号
  • 主机使用IP地址和端口号引导段到合适的套接字
    • 端口号：一个 16 位数字，从 0 到 65535
    • 0 至 1023：众所周知的端口号（HTTP:80, DNS: 53, IMAP: 143,FTP:20,21,SMTP:25,POP3:110）
    • 1024 至 65535：动态/专用端口

2.2 无连接多路分解

  • UDP 套接字由二元组标识：目标 IP 地址、目标端口号
  • 目标端口号相同但源 IP 地址/源端口号不同的 IP 数据报将定向到目标地址的同一套接字

2.3 有连接多路分解

  • TCP 套接字由四元组标识：源 IP 地址、源端口号、目的 IP 地址、 目的端口号

  • 源 IP 地址或源端口号不同将被导向两个不同的套接字

  • 携带原始连接建立请求的 TCP 段除外（欢迎套接字）

  • 服务器主机可同时支持多个 TCP 套接字：每个套接字都由自己的 四元组标识

  • 网络服务器为每个连接的客户端设置不同的套接字

    • 持久性HTTP使用同一套接字
    • 非持久性HTTP将为每个请求设置不同的套接字，频繁创建和关闭

  • 如今的高性能网络服务器只使用一个进程，并为每个新的客户端连接创建一个带有新连接套接字的新线程

3. 无连接传输：UDP

3.1 简介

  • “尽力而为 "服务，UDP 段可能：丢失、无序交付
  • Connectionless（含义）：
    • UDP 发送方和接收方之间无握手
    • 每个 UDP 段都独立于其他段处理
  • 应用场景：
    • 流媒体应用程序（丢失容忍，速率敏感）
    • DNS
    • SNMP （Simple Network Management Protocol）
    • RIP（Routing Information Protocol）
  • 通过 UDP 进行可靠传输：
    • 可能，但任务艰巨
    • 在应用层增加可靠性（增加确认和重传机制）
    • 特定应用程序的错误恢复

3.2 优点

  • 无拥塞控制：UDP 可以随心所欲地快速传输（非常适合实时应用）
  • 不建立连接（可减少延迟）
  • 简单：发送方和接收方无连接状态（支持更多活跃客户端）
  • 更小的头部（TCP：一般为 20 字节，UDP：8 字节）

3.4 校验和

  • 发送方：
    • 将段内容（包括头部）视为 16 位整数序列
    • 校验和：段内容中所有 16 位字之和的反码（1’s complement）
      • 先带循环进位累加，最高位溢出右移最低位，与最低位相加
      • 累加结果再求反码
    • 发送方将校验和值放入 UDP 校验和字段
  • 接收方：
    • 将段中所有 16 位字相加，包括校验和
    • 检查总和是否等于 111111111111111111
      • NO：检测到错误
      • YES：未检测到错误
  • 引入原因：
    • 其中一条链路可能使用不提供错误检查的链路层协议
    • 在路由器中存储段时可能会引入位错误
    • 提供错误检查作为安全措施
    • 注：UDP 不会采取任何措施从错误中恢复；有些 UDP 实现会直接丢弃损坏的数据段；有些则会将损坏的段交给应用程序并发出警告

4. 可靠数据传输原理

  • 服务抽象：可靠数据传输协议（ reliable data transfer protocol，rdt）

  • 只考虑单向（unidirectional）数据传输，但控制信息将双向流动

  • 使用有限状态机（finite state machines，FSM）来指明发送方、接收方。状态：当处于该状态时，下一个状态由下一个事件唯一决定。

4.1 rdt1.0

  • 底层信道完全可靠：
    • 无比特错误
    • 无数据包丢失
  • 发送器和接收器的独立 FSM：
    • 发送方向底层通道发送数据
    • 接收方从底层通道读取数据

4.2 rdt2.0

  • 底层信道可能会翻转数据包中的位

    • 仍假定数据包按顺序到达接收器，不会丢失
    • 校验和以检测位错误

  • 如何从错误中恢复：

    • （positive）acknowledgements（ACKs）：接收者明确告知发送方数据包
    • negative acknowledgements（NAKs）：接收方明确告知发送方数据包有错误（需要重传）
    • 发送方收到 NAK 后重传数据包

  • rdt2.0 中的新机制（对比 rdt1.0）

    • 错误检测
    • 反馈：从接收方到发送方的控制消息（ACK、NAK）
    • 重传

  • 停等协议（stop and wait）：发送方发送一个数据包，然后等待接收方回应

  • 缺点：ACK/NAK受损。解决方案分为三类：

    • 引入一种新型的发送方对接收方数据包，类似于人类所说的 ”你说什么？“
    • 添加校验位，以便发送方检测和恢复（对损坏的数据包有效，但不能解决丢失）
    • 正确收到ACK则发下个新数据，正确收到NAK则重发老数据，无法确认ACK或NAK（错误数据）则重发老数据，避免漏发
      • 由于接收方并不清楚刚刚发送的ACK或NAK是否被正确接受，因此接收方并不清楚收到的数据是重传（冗余）数据还是新数据
      • 解决：发送方为每个 pkt 添加序列号；对于简单的停止-等待协议，1 位序列号。

  • 与 rdt2.1 功能相同，仅使用 ACK
  • 接收方为正确接收的最后一个数据包发送 ACK，而不是 NAK

4.5 rdt3.0

  • 底层信道可能出现位错误、数据包丢失

  • 需要倒数定时器（countdown timer）：设置ACK 超时时间，如果在此时间内未收到 ACK 则重传，可能出现冗余数据包

  • 比特交替协议（alternating-bit protocol）

  • rdt3.0 性能不佳：停等协议限制了物理资源的使用

4.5.1 流水线协议

  • 流水线协议：发送方允许多个传输中、尚未确认的数据包
  • 必须增加序列号的范围
  • 已发送但未被确认的数据需要在发送方或接收方缓存
  • 两种实现：
    • Go-Back-N（GBN,回退N步）
    • selective repeat（SR, 选择重传）

4.5.2 GBN

  • 发送方允许流水线中存在N个未被确认的数据包
  • 接收方仅发送累计确认：如果有间隙，则不ACK该数据包
  • 发送方为最早未被确认数据维护一个定时器，一旦超时，重传所有数据包

发送方：

  • window size N：管道中连续未确认数据包的最大允许数量
  • send_base：最旧的未确认数据包的序列号
  • nextseqnum：最小未使用序列号
  • N 被称为窗口大小，GBN 协议又被称为滑动窗口协议（sliding-window protocol）
  • 为什么不允许数据报的数量是无限的？流量控制和拥塞控制
  • 数据包头中的 k 位序列号，范围 [0,2k-1]，TCP 有一个 32 位序列号字段（计算字节流中的字节，而不是数据包）
  • GBN 发送方必须对三类事件做出响应：
    • 上层调用：发送方首先检查窗口是否已满，未满-->创建并发送一个数据包，已满-->发送方直接将数据返回上层
    • 收到一个 ACK(n)：序列号n及其之前的所有报文被接收
    • 超时事件timeout(n)：重新传输窗口中所有序号大于等于n的数据包

接收方：

  • 总是为正确接收到的具有最高顺序序列的数据包发送 ACK
  • 丢弃所有乱序数据包

4.5.3 SR

  • 发送方允许流水线中存在N个未被确认的数据包
  • 接收方为每个数据包发送独立ACK
  • 发送方为每个未确认数据包均维护一个定时器，一旦超时，仅重 传未被确认数据包
  • 根据需要缓存乱序数据包，以便最终按顺序传送到上层
  • seq>=2*window

4.6 小结

  • 校验和 [first rdt 2.0]：用于检测传输数据包中的比特错误。
  • ACK [first rdt 2.0] ：接收方用于告知发送方已正确收到一个或一组数据包。根据协议的不同，确认可以是单独的，也可以是累积的。
  • NAK [first rdt 2.0]： 接收方用来告知发送方没有正确接收到数据包。带有未正确接收的数据包的序列号。
  • 序列号 [first rdt 2.1]： 用于对从发送方流向接收方的数据包进行顺序编号。接收到的数据包序列号中的间隙和重复，用于检测数据包的丢失和重复副本。
  • 定时器 [first rdt 3.0]：用于超时/重传数据包，原因可能是数据包（或其 ACK）在信道中丢失。接收方可能会收到数据包的重复副本。
  • 窗口、流水线 [first rdt 3.0+]：允许传输多个数据包但尚未确认，与停止和等待操作模式相比，可以提高发送方的利用率。 窗口大小可根据接收方接收和缓冲信息的能力或网络拥塞程度来设定，或两者兼而有之。

5. 面向连接传输：TCP

5.1 简介

  • 点对点（point-to-point）：
    • 一个发送方，一个接收方
    • 不支持一个发送方对多个接收方（多播）
  • 可靠、有序的字节流：没有 "信息边界"
  • 流水线：TCP 拥塞和流量控制设置窗口大小
  • 全双工数据（full duplex data）：
    • 同一连接中的双向数据流
    • MSS: maximum segment size 最大报文段长度 （传输层能承载的应用层最大数据长度，不包括传输层的头部）
  • 面向连接
  • 流量控制：发送方不会淹没接收方

5.2 段格式

  • 序列号：首个字节数据的编号

  • 确认号：接收方期待收到的字节数据的序列号（累计确认）。如：R 已收到 0-535、900-1000，R 的下一数据段 ACK字段包含 536。

处理乱序段：保留不按顺序排列的字节，并等待缺失的字节填补空白。

RTT与超时间隔：

  • SampleRTT：从段发送到收到 ACK 的时间
  • EstimatedRTT = (1- a)*EstimatedRTT + a*SampleRTT
    • 指数加权移动平均值(EWMA)
    • 过去样本的影响呈指数级下降
    • a=0.125（通常情况）
  • DevRTT = (1-a)*DevRTT + a*|SampleRTT-EstimatedRTT|
    • 用于估算：采样RTT偏离估计RTT的程度
    • 如果采样RTT值波动较小，那么DevRTT值也越小，反之亦然
    • a=0.25（通常情况）
  • TimeoutInterval = EstimatedRTT + 4*DevRTT

5.3 可靠数据传输

  • 流水线报文段
  • 累计确认
  • 单个重传定时器
  • 触发重传：超时事件、冗余ACK

5.3.1 发送方三个重要事件

  • 收到APP发送的数据
    • 用 seq # 创建段
    • seq段中第一个数据字节的字节流编号
    • 如果定时器尚未运行，则启动定时器
      • 定时器为最旧的未确认段
      • 超时间隔： TimeOutInterval
    • TimeoutInterval = EstimatedRTT + 4*DevRTT
  • 超时：重传导致超时事件的报文
    • TimeoutInterval 加倍，如果超时再次发生，TimeoutInterval 为 4 倍，除非收到段并更新了 EstimatedRTT。
  • 收到到ACK：如果 ACK先前没有ACK的段，更新已 ACK 的内容（累计确认）；如果仍有未ACK的段，则启动定时器。
    • TimeoutInterval = EstimatedRTT + 4*DevRTT

5.3.2 快速重传

  • 超时时间往往相对较长：重发丢失数据包前的延迟较长，增加端到端延迟。
  • 通过重复 ACK 检测丢失的段： 发送方经常发送许多连续数据段，如果数据段丢失，很可能会有许多重复的 ACK。
  • 快速重传：如果发送方收到对相同数据的3个冗余ACK，则发送序列号最小的未确认报文段。

5.4 流量控制

  • 流量控制：接收方控制发送方，因此发送方不会因传输过多、过快而溢出接收方的缓冲区
  • LastByteRead：应用层从缓存中读取的最新一个字节的编号
  • LastByteRcvd：到达接收缓存最新字节数据的编号
  • 不允许溢出分配的缓冲区LastByteRcvd – LastByteRead <= RcvBuffer
  • 接收缓存中空闲空间：rwnd = RcvBuffer – [LastByteRcvd – LastByteRead]
  • 接收方通过在接收方至发送方段的 TCP 标头中加入 rwnd 值来说明可用缓冲空间，发送方根据接收方的 rwnd 值限制未确认数据的数量。

5.5 连接管理

5.5.1 三次握手

  • 第 1 步：客户端主机向服务器发送 TCP SYN 段
    • 指定客户端初始序列号（随机）
    • 无数据
  • 第 2 步：服务器主机接收 SYN，回复 SYNACK 段
    • 服务器分配缓冲区和变量
    • 指定服务器初始序列号
    • 无数据
  • 第 3 步：客户端收到 SYNACK，用 ACK 段回复
    • 客户端分配缓冲区和变量
    • 可能包含数据（例如 HTTP）

5.5.2 四次挥手

  • 第 1 步：客户端系统向服务器发送 TCP FIN 控制段（FIN=1）
  • 第 2 步：服务器收到 FIN，回复 ACK。关闭连接，发送 FIN。(FIN=1)
  • 第 3 步：客户端收到 FIN，回复 ACK。 进入 "定时等待 "状态 - 将以 ACK 回应收到的 FIN。
  • 第 4 步：服务器收到 ACK。 连接关闭。
  • 注：S如果在Timeout内未收到C的ACK，则将重新发送FIN。C在Timed wait里面若再次收到来自S的FIN，将重发ACK并重新进入到Timed wait状态，确保S能进入到关闭

客户端：

  • FIN_WAIT_1：等待来自服务器的 ACK。
  • FIN_WAIT_2：等待来自服务器的 FIN 段。
  • TIME_WAIT：等待可能到达的 FIN，以防客户端的最终 ACK 丢失。

5.5.3 RST

当接收到端口号或源 IP 地址与任何正在进行的套接字（接收方）不匹配的 TCP 段时：

  • TCP：主机向源发送一个特殊的 RST 段
  • UDP：主机发送特殊的 ICMP 数据报

6. 拥塞控制原理

  • 拥塞：
    • 发送数据的来源太多、太快，网络无法处理
    • 与流量控制不同：接收方控制发送方，因此发送方不会溢出接收方的缓冲区
  • 表现：
    • 长时间延迟（路由器缓冲区排队）
    • 丢失数据包（路由器缓冲区溢出）

6.1 解决方法

  • end-end congestion control （端到端拥塞控制）
    • 没有来自网络的明确反馈
    • 根据终端系统观察到的丢失、延迟推测拥塞情况
    • TCP 采用此方法
  • network-assisted congestion control（网络辅助的拥塞控制）
    • 路由器向终端系统提供反馈
      • 用一个bit指示链路中的拥塞情况（SNA、DECbit、TCP/IP ECN、ATM、XCP）
      • 路由器为发送方提供明确的速率信息（ATM ABR）

6.2 ATM ABR

  • ATM (Asynchronous Transfer Mode)：异步传输模式，面向 虚电路的方式处理包交换

  • ABR: available bit rate

    • 弹性服务
    • 如果发送方路径轻负载：发送方应充分利用可用带宽
    • 如果发送方路径拥塞：发送方应抑制发送速率到最小保障的传输速率

  • RM (resource management) cells资源管理信元

    • 由发送方发送，中间夹杂数据信元（不是数据包）
    • 传送拥塞相关信息
    • 两种方式：
      • 直接网络反馈，即交换机直接反馈拥塞信息给发送方
      • 经由接收方的网络反馈，即RM抵达接收方后被修改指示信息然后反馈给发送方
    • RM 信元（不是数据包）中的两个比特由交换机（不是路由器）设置
      • NI：轻度拥塞
      • CI：严重拥塞
      • 接收方退回发送方的 RM 单元（CI/NI数据通常保持不变，下面特殊情况除外）
    • RM 单元中两个字节的 ER（显示速率）字段
      • 拥塞的交换机可能会降低小信元的 ER 值
      • 发送者的发送速率以及路径上可支持的最大速率
    • 每个数据单元中的一位 EFCI（显式转发拥塞指示）：拥塞的交换机设为1
      • 在接收到RM之前，若收到数据信元EFCI为1，则在收到RM后对其CI值进行修改

7. TCP拥塞控制

  • TCP 必须使用端到端拥塞控制，而不是网络辅助拥塞控制。IP 层不向终端系统提供明确反馈。
  • 限制发送速率作为感知网络拥塞的函数
    • 提高发送速率，感知到很少的拥塞
    • 降低发送速率，感知拥塞

7.1 如何限制发送速率

  • congestion window （cwnd,拥塞窗口）：
    • 对发送者在网络中发送流量的速率施加约束
    • LastBytesSent – LastByteAcked <= min{cwnd, rwnd}
  • 方便分析拥塞机制，假设接收缓冲区非常大，可以忽略 rwnd
    • LastBytesSent – LastByteAcked <= cwnd
  • 限制发送方中未被确认的数据量，间接限制发送速率
  • cwnd 是动态的，是感知网络拥塞的函数
  • TCP 发送速率：
    • 考虑丢失和数据包传输延迟可忽略不计
    • 大致为：发送一次 cwnd 字节，等待一个 RTT 收到 ACKS，然后发送更多字节
    • 调整 cwnd 以控制发送速率
    • \(rate≈\frac{cwnd}{RTT}bytes/sec\)

7.2 如何感知拥塞

  • 出现拥塞：丢包事件（超时或收到三个重复 ACK ）
  • 未出现拥塞：发送方收到ACK信息，意味着网络状态良好，增加拥塞窗口的大小
  • TCP 是自计时的（self-clocking）

7.3 拥塞控制算法

  • slow start (慢启动)
  • congestion avoidance (拥塞避免)
  • fast recovery (快速恢复)

7.3.1 慢启动

  • cwnd初始值=1MSS，发送速率≈MSS/RTT
  • 慢启动：当连接开始时，速率以指数方式增加，直到发生第一次丢失事件。
    • 一轮RTT，cwnd增加为原来的两倍
    • 发送方每收到一个ACK，cwnd增加1个MSS，一轮增加1个cwnd
  • 慢启动指数增长何时结束？
    • 超时事件
      • ssthresh=cwnd/2（ssthresh 是状态变量阈值）
      • cwnd=1MSS
      • 回到慢启动状态
    • cwnd >= ssthresh：进入拥塞避免状态
    • 3个冗余ACK
      • ssthresh=cwnd/2
      • cwnd=ssthresh+3*MSS
      • 执行快速重传，进入快速恢复状态

7.3.2 拥塞避免

  • 保守的方法：一轮RTT增加1个MSS，即发送方每收到一个ACK，cwnd增加1个MSS*(MSS/cwnd)
  • 拥塞避免的线性增长何时结束？
    • 超时事件
      • ssthresh=cwnd/2
      • cwnd=1MSS
      • 进入慢启动状态
    • 3个冗余ACK
      • ssthresh=cwnd/2
      • cwnd=ssthresh+3*MSS
      • 执行快速重传，进入快速恢复状态

7.3.3 快速恢复

  • 对于导致 TCP 进入快速恢复状态的丢失网段，每收到一个重复的 ACK，cwnd 的值就会增加 1 MSS（cwnd = cwnd +MSS）。
  • 超时事件
    • ssthresh=cwnd/2
    • cwnd=1MSS
    • 进入慢启动状态
  • 新的ACK
    • cwnd=ssthresh
    • 在清空 cwnd 之后进入避免拥塞状态
  • 建议但不强制要求快速恢复
    • [TCP Tahoe] (old version)：无快速恢复
      • 总是将 cwnd 设置为 1（超时或 3 个冗余ACK）
      • 已不再使用
    • [TCP Reno] (new version)：有快速恢复（3 个冗余ACK）
    • ssthresh = cwnd /2
    • cwnd = ssthresh +3*MSS (很可能网络未出现严重拥塞，因此可不从1开始。有些快速重传版本并未加3个MSS)

7.3.4 小结

  • 检测到所有超时事件，下一轮执行如下操作：

    • ssthresh=cwnd/2
    • cwnd=1MSS（网络拥塞很严重）
    • 进入慢启动状态

  • 检测到所有3次冗余ACK事件，下一轮执行如下操作：

    • ssthresh = cwnd /2（网络拥塞不严重）
    • cwnd = ssthresh +3*MSS
    • 进入快速恢复状态

  • 注意：如果当前cwnd不为偶数，计算新的ssthresh时一般向下取整。

7.4 TCP吞吐量

  • W：发生丢失的窗口大小（以字节为单位）
  • 平均窗口大小： \(\frac{3}{4}W\)
  • 平均吞吐量： \(\frac{3}{4}\frac{W}{RTT}bytes/sec\)
