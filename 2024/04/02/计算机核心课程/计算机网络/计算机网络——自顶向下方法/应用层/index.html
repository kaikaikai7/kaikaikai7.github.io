应用层

1. 网络应用原理

1.1 创建网络应用

  • 在不同终端系统上运行，通过网络通信；
  • 无需为网络核心设备编写软件。

1.2 应用架构

  • client-server（CS）：客户机-服务器架构
    • 传统应用，如E-mail、FTP、WWW
  • peer-to-peer (P2P) ：对等网络架构
    • 新型流量密集型应用，如IM(Instant Message),IPTV, Internet Phone, Live Streaming
  • hybrid of client-server and P2P：混合C/S和P2P架构

1.2.1 CS

  • 服务器：
    • 永远在线的主机
    • 永久 IP 地址(固定 IP)
    • Web、FTP、Telnet、电子邮件
    • 用于扩展的数据中心
  • 客户端：
    • 与服务器通信
    • 可能是间断地连接
    • 可能有动态 IP 地址
    • 不直接相互通信

1.2.2 P2P

  • 没有永远在线的服务器
  • 任意终端系统直接通信
  • 对等体向其他对等体请求服务，向其他对等体提供服务作为回报
    • self-scalability（可收缩性）：新的对等体带来新的服务能力，以及新的服务需求
  • 对等体间断地连接，可能改变 IP 地址
    • 复杂的管理
    • BitTorrent、eMule、Skpye、PPTV、Thunder
  • 成本低：通常不需要大量的服务器基础设施和服务器带宽

1.2.3 混合型

  • 集中式服务：客户端存在检测/定位
    • 用户上线时向中央服务器注册其 IP 地址
    • 用户联系中央服务器以查找好友的 IP 地址
  • 两个用户之间的聊天是 P2P 式的
  • 例子：Skype

1.3 进程通信

进程(process)：在主机内运行的程序。

  • 在同一主机内，两个进程使用进程间通信 (IPC) 进行通信（由操作系统定义
  • 在不同主机内，进程通过交换信息进行通信
    • 发起进程：创建并向网络发送信息
    • 接收进程：接收信息并进行响应
    • 定义：发起通信的进程被称为客户端，等待联系以开始会话的进程被称为服务器
    • 在网络应用程序中，客户端浏览器进程与网络服务器进程交换信息
    • 采用 P2P 架构的应用程序同时有客户端进程和服务器进程。在下面这个特定的通信会话中，A 请求 B 发送文件，A 是客户端，B 是服务器。

1.4 套接字

  • 进程通过一个称为套接字（socket）的软件接口向网络发送信息，并从网络接收信息。
  • 进程类似于房子，套接字类似于门
    • 发起进程将信息推出门外
    • 发起进程依靠门另一侧的传输基础设施将信息传送到接收进程的套接字
    • 应用层方面的控制，但对套接字传输层方面的控制却很少

1.5 寻址过程

  • 要接收信息，进程必须有标识符
  • IP地址区分不同主机，端口号区分相同主机不同进程
  • 标识符包括与主机上进程相关的 IP 地址和端口号
  • 端口号示例：
    • HTTP 服务器：80
    • 邮件服务器（SMTP）： 25
  • 向 gaia.cs.umass.edu 网络服务器发送 HTTP 信息：
    • P 地址：128.119.245.12
    • 端口号： 80

1.6 应用层协议

  • 定义了运行在不同端系统上的应用程序进程如何相互传递报文
    • 报文类型
    • 语法
    • 语义
    • 规则
  • 协议类型
    • open protocols (公开协议)：如HTTP, SMTP
    • proprietary protocols (私有协议)：如Skype, QQ, Thunder

1.7 应用程序需要的传输服务

  • data integrity(数据完整性)
    • 某些应用程序（如文件传输、网络交易）需要 100% 可靠的数据传输
    • 其他应用程序（如音频、视频）可以容忍一定程度的数据丢失
  • Timing(及时性)
    • 有些应用程序（如网络电话、互动游戏）需要低延迟才能有效运行
  • Throughput(吞吐量)
    • 带宽敏感应用：有些应用程序（如多媒体）需要最低限度的吞吐量才能有效
    • 弹性应用：可以利用它们获得的任何吞吐量（如电子邮件、文件传输、网络）
  • Security(安全性)
    • 加密、数据完整性

1.8 因特网传输协议服务

  • TCP 服务
    • 发送和接收方之间的可靠传输
    • 流量控制：发送方不会淹没接收方（避免接收方处理不过来）
    • 拥塞控制：当网络过载时对发送方进行节流（避免网络处理不过来）
    • 不提供：定时、最小吞吐量保证、安全
    • 面向连接：客户端和服务器进程之间需要进行连接设置
  • UDP 服务
    • 发送和接收方之间不可靠传输
    • 不提供：可靠性、流量控制、拥塞控制、定时、吞吐量保证、安全性或连接设置
  • TCP 和 UDP
    • 无加密 ；发送到套接字的信息以明文形式穿越因特网
    • SSL (Secure Sockets Layer, 安全套接层)
      • 提供加密连接、数据完整性、端点验证，以增强 TCP
      • 不是第三种互联网传输协议，在应用层实施
    • 今天的因特网传输协议不提供吞吐量或及时性服务，但通常可以为时间敏感的应用提供令人满意的服务（不保证，但能基本满足）

2. Web和HTTP

2.1 Web概览

  • 网页(web) 由基本 HTML 文件和多个引用对象组成
  • 对象可以是 HTML 文件、JPEG 图像、Java applet、音频文件
  • 每个对象都可以通过 URL 进行寻址

2.2 HTTP概览

  • HTTP: hypertext transfer protocol(超文本传输协议)
  • 网络的应用层协议
  • 客户端/服务器模式
    • 客户端：请求、接收（使用 HTTP 协议）和 "显示 "网络对象的浏览器
    • 服务器：网络服务器（使用 HTTP 协议）根据请求发送对象
  • 使用 TCP（底层传输协议）：
    • 客户端启动与服务器的 TCP 连接（创建套接字），端口为 80
    • 服务器接受来自客户端的 TCP 连接
    • 浏览器（HTTP 客户端）和 Web 服务器（HTTP 服务器）之间交换 HTTP 消息（应用层协议消息）
    • 关闭 TCP 连接
  • HTTP 是 "无状态 "协议（stateless protocol）：服务器不保留任何有关过去客户端请求的信息

2.3 HTTP连接

  • non-persistent connections (非持久连接)：每个请求/响应对通过单独的 TCP 连接发送

  • persistent connections (持久连接)：所有请求及其相应的响应都通过同一个 TCP 连接发送

  • HTTP 可以使用非持久连接和持久连接

    • 非持久 HTTP：

      • 通过一个 TCP 连接最多发送一个对象，随后连接关闭；下载多个对象需要多个连接

      • 用户可以通过配置浏览器来控制并行程度，使用并行连接可缩短响应时间

      • Round-trip Time (RTT, 往返时间)：一个小数据包（无传输延迟）从客户端到服务器再返回的时间，包括处理延迟、排队延迟和传播延迟

      • HTTP 响应时间 =2RTT+ 文件传输时间

        • 缺点：必须为每个请求对象建立和维护全新连接，增加操作系统开销 ； 每个对象的交付延迟超过两个 RTTs

    • 持久 HTTP：

      • 服务器在发送响应后保持连接打开
      • 同一客户端/服务器之间的后续 HTTP 消息通过同一连接发送  客户端在遇到引用对象时立即发送请求
      • 所有引用对象的 RTT 均为1

  • HTTP 的默认模式使用带有 pipelinling（流水线）的持久连接

    • Pipelinling：不等待已发送消息的应答，连续发送后面的消息。

2.4 HTTP消息

HTTP 消息由采用 ASCII 编码的多行文本构成，是服务器和客户端之间交换数据的方式。有两种类型的消息：请求（request）——由客户端发送用来触发一个服务器上的动作；响应（response）——来自服务器的应答。HTTP 请求和响应具有相似的结构，由以下部分组成：

  1. 一行起始行用于描述要执行的请求，或者是对应的状态，成功或失败。这个起始行总是单行的。
  2. 一个可选的 HTTP 标头集合指明请求或描述消息主体（body）。
  3. 一个空行指示所有关于请求的元数据已经发送完毕。
  4. 一个可选的包含请求相关数据的主体（比如 HTML 表单内容），或者响应相关的文档。

2.4.3 响应状态码

  • 200 OK：请求成功

  • POST method：表单输入信息在body字段
  • URL method：输入信息在URL中

2.4.5 方法类型

  • HTTP/1.0：GET、POST、HEAD（要求服务器在响应中不包含请求对象）

2.6 Cookie

  • 为识别用户，许多网站使用 cookie
  • 组成部分：
    • HTTP 响应信息中的 cookie 标头行
    • 下一个 HTTP 请求信息中的 cookie 标头行
    • 保存在用户主机上的 cookie 文件，由用户的浏览器管理
    • 网站的后端数据库
  • 用途：授权、购物车、推荐
  • 保存状态
    • 协议端点：在多个事务中保持发送方/接收方的状态
    • cookie：使 http 消息携带状态，在无状态 http 上创建用户会话层

2.7 Web caches

  • 目标：满足客户端请求，而不涉及源服务器
  • 用户设置浏览器：通过代理服务器访问 Web
  • 浏览器向代理服务器发送所有 HTTP 请求
    • 缓存中已有对象：缓存返回对象
    • 否则缓存向源服务器请求对象，然后将对象返回给客户端
  • 代理服务器同时充当客户端和服务器
  • 使用原因：
    • 减少客户端请求的响应时间（如在线游戏）
    • 减少机构接入链路到因特网的流量（如 html 文件缓存）

2.8 内容分发网络

  • Content Distribution Networks (CDNs)

  • 网络缓存在互联网上发挥着越来越重要的作用

  • 在整个互联网上安装了许多地理位置分散的缓存，从而将大部分流量本地化。

2.9 条件GET

在网络缓存中的对象副本可能是陈旧的，条件获取（Conditional GET）允许网络缓存检查对象是否是最新的。

3. 文件传输协议

  • FTP: File Transfer Protocol （文件传输协议）

  • 客户机/服务器模式

    • 客户机：发起传输的一方
    • 服务器：远程主机

  • 使用 TCP协议作为底层传输协议

  • 客户端通过控制连接进行授权，发送命令

  • 服务器收到文件传输命令后，向客户端打开第二个 TCP 数据连接

  • 控制连接在用户会话期间保持打开，传输一个文件后，服务器关闭数据连接，打开另一个 TCP 数据连接以传输另一个文件

  • “out of band” 带外：有两个通道同时进行交互（http,smtp:”in-band”）

  • 有状态协议：如当前目录，更早的验证

4. 电子邮件

  • 用户代理

    • 又称 "邮件阅读器"
    • 撰写、编辑、阅读、回复、转发、保存邮件
    • 如 Outlook、Thunderbird、iPhone 邮件客户端
    • 发送、接收的邮件存储在服务器上

  • 邮件服务器

    • 邮箱包含用户接收的邮件
    • 发送（待发）邮件信息的信息队列
    • 邮件服务器之间发送邮件信息的 SMTP 协议
      • 客户端：发送邮件的服务器
      • 服务器：接收邮件的服务器

  • 简单邮件传输协议（SMTP）

    • 使用 TCP 协议， 25号端口（服务器），持久连接，有状态协议
    • 直接传输：发送服务器（SMTP 客户端）到接收服务器（SMTP 服务器）
    • 命令/响应交互（如 HTTP、FTP）
      • 命令：ASCII 文本
      • 响应：状态代码和短语
    • SMTP 服务器使用 CRLF.CRLF 确定信息结束

5. DNS

  • Domain Name System：域名系统
  • 一个由分层的DNS服务器实现的分布式数据库
  • 一个使主机能查询分布式数据库的应用层协议
  • 下层协议：UDP
  • 端口：53

5.1 DNS服务

通常由其他应用层协议（包括 HTTP、SMTP 和 FTP）使用

  • 域名到IP地址转换
  • 主机别名：获取主机别名对应的规范主机名及主机IP地址
  • 邮件服务器别名：获取邮件服务器别名对应的规范主机名及主机IP地址
  • 负载分配

5.2 分布式层次数据库

5.3 本地DNS服务器

  • 不属于（以上）层次结构
  • 每个 ISP都有一个，也称为 "默认名称服务器"
  • 当主机连接到 ISP 时提供一个或多个本地 DNS 服务器的 IP 地址
  • 当主机进行 DNS 查询时，查询会被发送到其本地 DNS 服务器：
    • 本地缓存有最近的名称到地址转换对（但可能已经过期！）
    • 充当代理，将查询转发到层次结构中
  • 一旦DNS服务器获得映射，它就会缓存映射；缓存可能过时
  • TLD 服务器通常缓存在本地DNS服务器中

5.4 DNS查询

迭代查询：被联系的服务器回复要联系的服务器名称（常用的模式）

递归查询：将名称解析的重任交给联系的名称服务器；层次结构中上层负担重

5.5 DNS记录

DNS：存储资源记录 (RR) 的分布式数据库

  • Type=A ：标准的主机名到IP映射（www.bar.foo.com, 145.37.93.126, A）
  • Type=NS：域的权威DNS服务器（foo.com, dns.foo.com, NS）
  • Type=CNAME：别名对应的规范主机名（foo.com, relay1.bar.foo.com, CNAME）
  • Type=MX：别名对应的邮件服务器的规范主机名（foo.com, mail.bar.foo.com, MX）

5.6 DNS报文

DNS 报文分为 请求 和 应答 两种，结构是类似的，大致分为五部分：

  • 头部（ header ），描述报文类型，以及其下 4 个小节的情况；
  • 问题节（ question ），保存查询问题；
  • 答案节（ answer ），保存问题答案，也就是查询结果；
  • 授权信息节（ authority ），保存授权信息；
  • 附加信息节（ additional ），保存附加信息；

头部分为 6 个字段：

  • 标识（ identifier ），一个 16 位的 ID ，在应答中原样返回，以此匹配请求和应答；
  • 标志（ flags ），一些标志位，共 16 位；
  • 问题记录数（ question count ），一个 16 位整数，表示问题节中的记录个数；
  • 答案记录数（ answer count ），一个 16 位整数，表示答案节中的记录个数；
  • 授权信息记录数（ authority record count ），一个 16 位整数，表示授权信息节中的记录个数；
  • 附加信息记录数（ additional record count ），一个 16 位整数，表示附加信息节中的记录个数；

标志字段中的各个标志位：

  • QR 位标记报文是一个查询请求，还是查询应答；
    • 0 表示查询请求；
    • 1 表示查询应答；
  • 操作码（opcode）占 4 位，表示操作类型：
    • 0 代表标准查询；
    • 1 代表反向查询；
    • 2 代表服务器状态请求；
  • AA 位表示 权威回答（ authoritative answer ），意味着当前查询结果是由域名的权威服务器给出的；
  • TC 位表示 截短（ truncated ），使用 UDP 时，如果应答超过 512 字节，只返回前 512 个字节；
  • RD 位表示期望递归（recursion desired），在请求中设置，并在应答中返回；
    • 该位为 1 时，服务器必须处理这个请求：如果服务器没有授权回答，它必须替客户端请求其他 DNS 服务器，这也是所谓的 递归查询 ；
    • 该位为 0 时，如果服务器没有授权回答，它就返回一个能够处理该查询的服务器列表给客户端，由客户端自己进行 迭代查询 ；
  • RA 位表示可递归（ recursion available ），如果服务器支持递归查询，就会在应答中设置该位，以告知客户端；
  • 保留位，这 3 位目前未用，留作未来扩展；
  • 响应码（response code）占 4 位，表示请求结果，常见的值包括：
    • 0 表示没有差错；
    • 3 表示名字差错，该差错由权威服务器返回，表示待查询的域名不存在；

  • 重定向攻击：中间人、DNS投毒
  • 利用DNS进行DDoS

6. P2P应用

6.1 文件分发

  • 客户机-服务器架构：服务器必须按顺序上传N 份文件副本，每个客户端必须下载文件副本。 \[ D_{c-s} \geq max\{\frac{NF}{u_s},\frac{F}{d_{min}}\} \]

  • 对等网络架构：服务器必须上传至少一份副本，每个客户端必须下载文件副本，总计必须上传N 份文件副本。 \[ D_{p2p} \geq max\{\frac{F}{u_s},\frac{F}{d_{min}},\frac{NF}{u_s+\sum_{i=1}^N u_i}\} \]

6.2 BitTorrent

BitTorrent 协议是架构于 TCP/IP 协议之上的一个 P2P 文件传输协议，处于 TCP/IP 结构的应用层。如果有多个下载者并发的下载同一个文件，则每个下载者也同时为其它下载者上传文件，这样，文件源可以支持大量的用户进行下载，而只带来适当的负载的增长。BitTorrent 协议把提供下载的文件虚拟分成大小相等的块，块大小必须为 2k 的整数次方（由于是虚拟分块，硬盘上并不产生各个块文件），并把每个块的索引信息和 Hash 验证码写入 .torrent 文件（即种子文件，也简称为“种子”）中，作为被下载文件的“索引”。 下载者要下载文件内容，需要先得到相应的 .torrent 文件，然后使用 BT 客户端软件进行下载。

  • 请求块：稀少资源优先
  • 发送块： tit-for-tat（以牙还牙）
    • 向当前向其发送信息块速率最高的 4 个对等节点发送信息块 ，其他对等节点被阻塞，每 10 秒重新评估前 4 个对等节点
    • 每过 30 秒，随机选择另一个对等点，开始发送数据块

7. 套接字编程

  • 套接字：应用过程（房屋）和终端传输协议之间的门
  • 针对两种传输服务的两种套接字类型：
    • UDP：无连接、不可靠数据报、不保证交付
    • TCP：面向连接、可靠、面向字节流

7.1 UDP

  • 发送数据前不握手
  • 发送方明确地为每个数据包附加 IP 目标地址和端口
  • 接收方从收到的数据包中提取发送方 IP 地址和端口

7.2 TCP

  • 服务器必须准备就绪

    • 服务器进程必须首先运行
    • 服务器必须创建了欢迎客户联系的套接字（门）

  • 客户端通过以下方式联系服务器

    • 创建 TCP 套接字，指定服务器进程的 IP 地址和端口号

    • 当客户端创建套接字时，客户端 TCP 与服务器 TCP 建立连接

    • 当客户端联系服务器时，服务器 TCP 为服务器进程创建新的套接字，以便与特定客户端通信
